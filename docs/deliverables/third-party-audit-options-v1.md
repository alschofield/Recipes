# Third-Party Audit Options (V1)

Practical ways to source qualified security/legal reviewers.

## Security Audit Sourcing Options

### Option A - Dedicated security consultancy

- Best for: end-to-end API/web/mobile assessment with formal report.
- Typical output: severity-ranked findings + re-test.
- Cost profile: medium-high.

### Option B - Fractional AppSec consultant

- Best for: architecture/threat-model review + targeted penetration checks.
- Typical output: prioritized remediation plan and release checklist signoff support.
- Cost profile: medium.

### Option C - Bug bounty / managed crowdsourced testing

- Best for: broad adversarial coverage after baseline hardening.
- Typical output: externally reported vulnerabilities over time.
- Cost profile: variable.

## Legal/Compliance Sourcing Options

### Option A - Tech/privacy counsel (fixed package)

- Best for: privacy policy + store declarations + launch legal memo.
- Cost profile: medium-high.

### Option B - Fractional product counsel

- Best for: ongoing support while iterating features post-launch.
- Cost profile: medium.

### Option C - Specialized OSS/compliance advisor

- Best for: attribution, license compatibility, and open-source notice obligations.
- Cost profile: medium.

## Suggested Selection Criteria

- Demonstrated experience with mobile + API + web audits.
- Clear remediation deliverables and retest policy.
- Ability to work from pre-built evidence pack docs.
- Turnaround time compatible with your release target.
- Fixed-scope pricing where possible.

## Recommended Engagement Sequence

1. Share pre-audit reports and briefs from `docs/deliverables/`.
2. Run security review first (catch technical blockers early).
3. Run legal/compliance review in parallel once policy/store drafts are stable.
4. Update `docs/ops/v1-launch-blocker-evidence.md` with reviewer outputs.
